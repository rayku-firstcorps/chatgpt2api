# ChatGPT2API 业务逻辑 Review 与产品计划

## 1. 产品定位

ChatGPT2API 当前不是一个完整的通用 OpenAI API 代理，而是围绕 ChatGPT 官网能力逆向封装出来的自托管图片生成与图片编辑产品。核心价值是把官网图片能力包装成 OpenAI 兼容接口，并提供在线画图、号池管理、图片资产管理、配置管理和部署能力。

从产品视角看，项目有两个主要使用场景：

1. API 接入场景：用户通过 `/v1/images/generations`、`/v1/images/edits`、`/v1/chat/completions`、`/v1/responses` 接入 Cherry Studio、New API 或自建应用。
2. 自带工作台场景：用户在 Web 端完成文生图、图生图、多参考图编辑、历史回看、图片下载和图片资产整理。

项目应继续把“图片工作流稳定可用”作为主线，文本、Anthropic、工具调用等兼容能力应被明确标记为附属能力或实验能力，避免用户误以为这是完整通用代理。

## 2. 当前业务闭环

### 2.1 鉴权与角色

系统存在两类身份：

- 管理员密钥：来自 `config.json` 或 `CHATGPT2API_AUTH_KEY`，可访问管理后台和所有 `/api/*` 管理接口。
- 普通用户密钥：由管理员在设置页创建，可调用图片与文本兼容接口，但不能管理账号池、配置、图片资产和日志。

这个模型满足单机自托管的基本权限隔离，但目前还没有更细的配额、并发、审计和用户维度成本控制。

### 2.2 图片 API 链路

同步图片接口流程：

1. 用户请求 `/v1/images/generations` 或 `/v1/images/edits`。
2. 后端校验身份、解析请求体、执行本地敏感词和可选 AI 审核。
3. 从账号池选取可用账号并占用图片并发槽位。
4. 调用 ChatGPT 官网会话流，识别图片工具事件、会话 ID、文件 ID 或沉淀资源 ID。
5. 拉取上游图片，保存到本地或 WebDAV，并按 OpenAI 兼容结构返回 `b64_json` 和/或 `url`。
6. 根据结果更新账号成功/失败次数、剩余额度、限流状态。

异步图片任务流程：

1. Web 工作台把每一张图片拆成一个 `client_task_id`。
2. 调用 `/api/image-tasks/generations` 或 `/api/image-tasks/edits` 提交后台线程任务。
3. 前端轮询 `/api/image-tasks?ids=...` 获取状态。
4. 成功后把图片 URL 写入本地会话历史。

同步 API 支持 `n=1-4`；Web 工作台允许最多 100 张，但实际是拆成多个单图异步任务执行。这是合理的交互设计，但需要在产品文档中明确，避免把 Web 端 100 张理解为单 API 请求支持 100 张。

### 2.3 号池管理链路

账号池是图片能力可用性的核心。当前逻辑包括：

- 添加、导入、删除、导出账号。
- 刷新账号邮箱、类型、额度、恢复时间等信息。
- 生图时按轮询选择可用账号。
- 每个账号通过 `image_account_concurrency` 控制并发占用。
- 成功后扣减额度；额度归零后标记限流。
- Token 失效时可标记异常或自动移除。
- 限流账号由后台 watcher 定期刷新恢复。

当前业务假设是“账号池总额度 = 服务可用产能”。这个假设对图片生成很实用，但还缺少面向用户密钥的配额隔离，所以多个普通用户共享同一号池时可能互相抢占额度。

### 2.4 Web 工作台链路

Web 端已经形成较完整的图片工作台：

- 登录后按角色跳转。
- 普通用户进入图片页，管理员可进入账号、设置、日志、图片管理等页面。
- 图片页支持文生图、上传参考图后进入编辑模式、多张任务队列、历史对话、继续编辑、重试、重新生成。
- 历史记录主要保存在浏览器本地，任务状态保存在服务端 `image_tasks.json`。
- 管理员可查看可用额度，普通用户只看到不可用占位。

产品体验上，图片工作台已经能独立使用；但它和 API 侧的能力说明仍有不一致，例如模型选择、批量数量、历史持久化边界和失败恢复策略。

### 2.5 管理与运维链路

系统已经提供：

- 全局代理配置与测试。
- 敏感词与可选 AI 审核。
- 图片本地存储、缩略图、WebDAV 同步。
- 日志查看与清理。
- 配置备份到 Cloudflare R2。
- JSON、SQLite、PostgreSQL、Git 多种元数据存储后端。
- CPA、sub2api、自动注册等账号来源集成。

这些能力对自托管用户很有价值，但配置项较多，需要明确默认推荐路径，否则新用户容易不知道最小可用配置是什么。

## 3. 主要业务问题

### P0：产品边界需要重新收敛

README、功能状态和接口实现中同时出现图片 API、文本 Chat Completions、Responses、Anthropic Messages、自动注册、账号导入、备份、图片资产管理等能力。能力很多，但主线不够突出。

建议把产品公开叙事收敛为：

- 主能力：OpenAI 兼容图片生成/编辑 API。
- 配套能力：Web 图片工作台、号池管理、图片资产管理、部署运维。
- 实验能力：文本对话、Anthropic Messages、工具调用适配、自动注册。

这样可以降低用户误用预期，也能指导后续开发优先级。

### P0：普通用户没有配额和并发隔离

当前普通用户密钥只做身份识别，不限制每个用户的请求量、并发数和可用额度。多个用户共享同一个号池时，一个用户可以耗尽全部图片额度。

建议新增用户密钥级别的：

- 每日/每月图片张数配额。
- 最大并发任务数。
- 是否允许图生图。
- 可用模型范围。
- 用量统计和最后调用时间。

### P1：多图能力存在两套产品语义

同步 API 的 `n` 限制是 `1-4`，Web 工作台最多可输入 100，但通过多个异步任务实现。两者都合理，但如果不说明，会造成 API 用户和 Web 用户预期不一致。

建议：

- API 文档明确 `/v1/images/*` 单请求最多 4 张。
- Web 文案明确“批量生成会拆分为多个任务”。
- 异步任务接口增加批量提交能力，避免前端一次创建大量 HTTP 请求。

### P1：任务队列能力偏轻，缺少全局调度视图

异步任务当前由进程内线程执行，任务状态写入 JSON。服务重启后未完成任务会被标记失败。这个设计简单可靠，但不适合大量任务、长时间任务和多实例部署。

建议短期继续保留当前实现，但增加：

- 全局任务列表和运行中任务数。
- 任务取消能力。
- 单用户最大排队数。
- 任务超时状态和失败原因分类。
- 管理员手动重试失败任务。

中期再考虑 SQLite/PostgreSQL 队列或外部任务队列。

### P1：账号调度策略还不够产品化

目前调度以轮询、额度、状态、账号并发为主，缺少更细的策略：

- 不同模型能力没有完整映射到账号级别。
- Plus/Team/Pro、Codex、生图额度未知账号的策略需要更明确。
- 失败重试没有区分网络失败、内容拒绝、账号限流、Token 失效、上游超时。
- 没有显示每个账号的近期成功率和平均耗时。

建议把账号状态从“正常/限流/异常/禁用”扩展为更可运营的维度：可用性、剩余额度、能力标签、近期错误、恢复时间、并发占用。

### P1：安全与合规提示需要进入产品流程

README 有免责声明和风险提示，但 Web 使用流程中缺少更明确的安全边界。考虑到项目涉及逆向能力和账号池，管理后台应在关键位置提示风险。

建议：

- 首次登录展示风险确认。
- 账号导入、自动注册、公开部署配置页显示风险提醒。
- 普通用户密钥创建时提示不要外发管理员密钥。
- 增加公网部署检查项：默认密钥、CORS、代理、日志敏感信息、图片公开 URL。

### P2：文档和实际实现存在不一致

`feature-status.en.md` 中部分状态与代码和 README 存在偏差，例如 Anthropic Messages 代码已有实现，但功能状态写的是待实现；图片尺寸参数写待实现，但当前通过提示词方式支持比例。该类不一致会直接影响用户接入判断。

建议把文档拆分为：

- `docs/api.md`：只写稳定接口。
- `docs/feature-status.md`：按“稳定/实验/规划/废弃”维护。
- `docs/operations.md`：部署、存储、备份、代理、安全检查。
- `docs/product-plan.md` 或当前 `plan.md`：产品路线图。

## 4. 产品路线图

### 阶段一：稳定图片核心链路

目标：让图片 API 和 Web 工作台成为可信赖的主功能。

计划：

- 修正文档状态不一致，明确稳定能力与实验能力。
- 完善图片错误分类：内容拒绝、无可用账号、账号限流、Token 失效、上游超时、网络失败。
- 在 Web 工作台展示更清晰的失败原因和可操作建议。
- 给异步任务增加取消、重试和管理员视图。
- 增加批量任务提交接口，前端批量生成不再一次发起大量请求。

验收标准：

- 用户能从 README 判断项目主能力和限制。
- 常见失败都能返回可理解错误。
- 100 张批量生成不会造成前端请求风暴。
- 服务重启后的任务状态有明确说明和恢复策略。

### 阶段二：用户级运营能力

目标：支持多人共享时的可控使用。

计划：

- 普通用户密钥增加配额、并发、模型范围、图生图权限。
- 日志增加用户维度聚合：调用次数、成功率、图片张数、失败原因。
- 图片任务和图片资产增加 owner 维度筛选。
- 管理员可暂停某个用户密钥的运行中任务。

验收标准：

- 管理员可以限制某个用户每天最多生成多少张图。
- 一个用户不能耗尽所有并发槽位。
- 管理后台可以按用户追踪消耗和失败。

### 阶段三：账号池调度升级

目标：提高成功率和账号利用效率。

计划：

- 为账号增加能力标签：官网生图、Codex 生图、文本、Plus/Team/Pro、额度未知。
- 调度时按模型和能力标签选择账号。
- 引入失败冷却策略，网络错误短冷却，限流长冷却，Token 失效移除或标异常。
- 展示账号级成功率、平均耗时、近 24 小时失败原因。
- 支持账号分组，用户密钥可绑定指定账号组。

验收标准：

- 不同模型请求不会错误分配到不支持的账号。
- 连续失败账号会自动降权或冷却。
- 管理员可以看出哪些账号质量差。

### 阶段四：部署与数据可靠性

目标：让自托管用户更容易长期运行。

计划：

- 明确推荐部署方式：Docker Compose + JSON/SQLite 起步，PostgreSQL 用于多人和长期运行。
- 补充配置健康检查页：auth-key、存储、代理、备份、图片目录、WebDAV。
- 为图片任务迁移到统一存储后端预留接口，减少 JSON 文件单点问题。
- 备份支持恢复向导，而不仅是下载备份。
- 增加日志脱敏策略说明和配置。

验收标准：

- 新用户按文档 10 分钟内完成最小可用部署。
- 管理员能在页面上看到关键配置是否健康。
- 备份不仅能生成，也能指导恢复。

## 5. 建议的产品指标

核心指标：

- 图片请求成功率。
- 图片平均生成耗时和 P95 耗时。
- 每日生成图片数量。
- 账号池可用账号数和总剩余额度。
- 账号限流率、异常率、Token 失效率。
- Web 工作台任务失败率。

管理指标：

- 普通用户密钥数量。
- 每个用户图片消耗。
- 单用户并发峰值。
- 备份成功率。
- 图片存储占用和清理数量。

体验指标：

- 首次部署成功率。
- 首次导入账号到首次成功生图耗时。
- 失败后用户重试成功率。

## 6. 近期执行清单

优先建议按以下顺序推进：

1. 修正文档不一致，重写功能状态为“稳定/实验/规划”。
2. 增加用户密钥级配额和并发限制的数据结构。
3. 给异步任务增加取消、批量提交和管理员任务列表。
4. 改造错误码和错误文案，让前端能区分可重试与不可重试。
5. 增加账号能力标签和模型到账号能力的调度规则。
6. 在设置页增加部署健康检查和风险确认。
7. 补充端到端测试：账号池无额度、Token 失效、上游超时、多图任务、普通用户配额耗尽。

## 7. 结论

ChatGPT2API 当前已经具备完整的图片 API 与 Web 图片工作台闭环，真正的产品瓶颈不在“能不能生图”，而在多人共享、任务调度、错误可解释性、账号运营和文档边界。下一阶段应把图片主链路做稳，把普通用户运营能力补齐，再逐步增强账号池调度与长期部署可靠性。
